## Archive Markdown

- 目标长度：**<= 3400 中文字符**。
- 正文是**上限约束**：最多 **8 条**，其中科技最多 **5 条**、学术最多 **3 条**。
- 默认目标：**6-8 条**。
- 默认结构：**至少 3 条可引流热点** + **至少 2 条独家深度/非显而易见判断**，其余位置再补当天最强信号。
- 不把上限当 KPI，但也不要过早停在 4 条。除非当天真的薄到不够，否则应补足到 **6 条以上**。
- 如果明显少于上限，开头建议明确写：`今日信号较少，仅收录 X 条。`
- 开头必须先写 `硅谷风向词`，放 **2-3 个**近期 X 上真实存在的概念词。
- `硅谷风向词` 必须是概念 / 框架，不要只写公司名或产品名。
- `硅谷风向词` 时间窗口：优先 **7 天内**，最多 **14 天**。
- 排除已经连续流行 **30+ 天** 的常识词。
- 排除只被 **1-2 个账号**孤立使用的词。
- 每个风向词都要带 1 个可追溯的 X URL，方便后续追问出处。
- 每个风向词解释尽量 **<= 38 个中文字符**。
- 每条必须包含：**标题 + 内容 + 解读 + 1 个原文链接 + 1 条可核验引用**。
- 每条都必须回答两个问题：
  - **发生了什么变化**：不是“谁发了什么”，而是“产品 / 工作流 / 技术约束哪里变了”。
  - **为什么现在值得看**：不是泛泛而谈“很重要”，而是点出对 builder / researcher / operator 的直接影响。
- 每条都必须带一个**专业角度**，至少满足其一：
  - 机制变化
  - 工作流变化
  - 成本 / 分发 / 权限结构变化
  - 评测口径或研究边界变化
- 每条还必须至少满足其一：
  - **可引流热点**：普通读者/产品人一眼能感到 relevance，容易引发转发与讨论；
  - **独家深度**：不是重复 headline，而是提供别人没讲透的判断、解释或边界条件。
- 明确排除“烂大街条目”：
  - 单纯融资新闻，除非它改写了产品能力边界或平台格局
  - 纯公司 PR 口径，除非能指出非显而易见的第二层影响
  - 没有一手细节的“某模型/某功能上线了”
  - 只是在中文圈传播很热，但英文一手和技术圈没有新信息增量的泛热点
  - RT / 转述型社交贴、泛问答、订阅攻略、经验分享、圈内 meta discourse
- 如果是研究条目，`内容` 不能只写“提出了什么”；必须点出 **方法变化** 或 **实验结果中最关键的 delta**。
- 如果是研究条目，`解读` 不能只写“很重要”；必须点出 **限制 / 适用边界 / 未来落地路径** 之一。
- 标题尽量 **<= 30 个中文字符**。
- `内容` 尽量 1-2 句，**<= 120 个中文字符**。
- `解读` 尽量 1 句，**<= 60 个中文字符**。
- `解读` 不能重复 `内容`；它必须是作者视角，不是改写新闻稿。
- 写法优先像**投研 memo / 产品策略笔记**，不要像“媒体资讯播报”。
- `原文链接` 优先第一手（release/blog/paper/repo/官方公告/原作者贴），聚合站只能做补充。
- 引用必须短、可在原文直接找到；找不到引用就丢掉该条。
- 如果只有单一直接一手来源，但内容足够重要且可核验，允许入选，并在条目末尾标 `状态：[单源]`。
- `【未证实】` 只用于争议性或 rumor-like 内容，不要和 `[单源]` 混用。
- 不得编造：指标、价格、发布日期、benchmark 分数、引用内容。

## Telegram Delivery

- 额外产出一段“Telegram 可直接发送”的摘要文本。
- 目标长度：**<= 2000 中文字符**。
- 如果当日明显少于上限，Telegram 版也要保留薄信号说明：`今日信号较少，仅收录 X 条。`
- 开头先用 1-2 行概括：`硅谷风向词：词1 / 词2 / 词3`。
- 只保留当日最终入选条目，不要额外加标题或一句话总结。
- 每条最多 3 行：`emoji + 标题`、`内容：...`、`解读：...`。
- Telegram 正文不要放每条链接。
- 只允许放 **1 个** 链接：选当天最热点那条的 primary link，单独放在全文最后一行。
- 这个链接必须是最后一个非空行，后面不要再加任何字，这样 Telegram 才会稳定显示预览。
- 最终对话回复只输出这段 Telegram 文本，不要附加路径、调试信息、完成说明。
- Telegram 版也要保留**判断密度**，不能退化成“今天发生了几件事”的流水账。
- 如果当天还产出了视频，Telegram 交付顺序固定为：**封面图 -> 每日视频 -> Telegram 文本**。
- 每日视频流水线内部只认 `final/video.mp4`；同时必须生成一个标题命名副本：`标题【Lumi的AI速递第N期】.mp4`。
- Telegram 发送视频时默认用 `sendVideo`，并由 `publish/telegram-send.json.video_file` 指向 `final/video.mp4`。

## Cover Delivery

- 配图（Telegram）不要指望 Markdown 内联图能显示，改为单独发送媒体消息。
- 优先发 **1 张 16:9 拼贴封面图**（基于最终条目生成，优先使用真实参考图做组装）。
- 这张封面图必须来自 plus-media-factory 的正式封面链路；不要拿 `build/video/slides/cover.png` 充当日报封面。
- 先生成 `/Users/dystopia/Desktop/AI-Daily-Reports/YYYY-MM-DD/tech-daily-YYYY-MM-DD.cover-copy.json`，其中必须包含：
  - `marketing_headline`：每天 1 条用于营销传播的主标题，通常 8-16 个中文字符，必须真实、不造势、不偏离当日报告。
  - `subhead`：主标题下方的支持性副标题。
  - `left_lines` / `right_lines`：2-4 条短 cover lines。
- 抓封面图时，原始文章页优先于 `x.com` / 社交讨论链接；社交链接负责提供信号，不应默认抢封面素材配额。
- `arXiv` 页面若只有通用 logo，可接受 PDF 首图预览作为右下贴纸或辅助素材，但不要把它当主背景。
- 科技日报封面默认参考最初那张短视频封面风格：完整背景图、左上新闻块、大黄黑边营销主标题、白色粗体副标题、右下贴纸位。参考图只用于布局，不得直接混入最终成片。
- 不得保留明显浏览器 UI、二维码页、纯图表页、深色评测分数图、假条码、假时间戳或 fake mascot。
- 如果没有一等 `browser` / `message` 工具，就改用 OpenClaw CLI fallback：
  - `openclaw browser --browser-profile openclaw ...`
  - `openclaw message send --channel telegram --target <chat-id> --media <file>`
- 如果视频也存在，第二条 Telegram 媒体消息应发送 `/Users/dystopia/Desktop/AI-Daily-Reports/YYYY-MM-DD/final/video.mp4`，再发送 `telegram.txt` 里的文本正文。
- 若发送链路支持 Telegram Bot API，优先按 `sendPhoto -> sendVideo -> sendMessage` 执行。
- 若明确要多图，再逐条发送配图 + 单条 caption。
- 拿不到稳定图片就跳过，不要让整次日报失败。
